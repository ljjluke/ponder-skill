# MCTS-TD Planner — 多方案独立推演决策引擎

> **核心能力**：当有多个候选方案时，每个方案都在内部独立走完完整的执行路径推演（不实际执行），然后汇总比较选出最优方案，最后才执行。

此 Skill 将 **MCTS（蒙特卡洛树搜索）** 和 **TDL（时序差分学习）** 的算法思想注入到 Claude 的推理过程中。它不是实际的数值计算引擎，而是**将算法的核心决策逻辑转化为结构化的推理规则**，让 Claude 在面对复杂决策时，能够像拥有"内部推演棋盘"一样，系统地、逐个地推演每个候选方案，再综合比较。

> 本项目受 [hrpan/tetris_mcts](https://github.com/hrpan/tetris_mcts) 启发，该项目的 MCTS-TD 混合架构在俄罗斯方块游戏中取得了超越人类水平的成绩。

## 核心决策流程

```
┌──────────────────────────────────────────────────────────────────────┐
│               第0步: 需求约束收集（先问清楚，再动手）                  │
│  约束检测清单: 技术栈/依赖/架构/政策/性能/安全/时间/隐式               │
│  → 硬约束直接排除不合规方案 → 软约束影响匹配度评分                     │
│  → 信息不全就暂停问用户，不假设                                       │
└──────────────────────────┬───────────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────────────┐
│               第1阶段: 领域认知与方案生成                               │
│  检测领域熟悉度 → 如果不熟: 查资料(项目代码/文档/知识图谱)               │
│  → 带着约束清单生成方案 → 每个方案标注来源依据和约束遵守情况            │
│  → 超过5个时粗筛保留 top-3~5                                          │
└──────────────────────────┬───────────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────────────┐
│                        第2阶段: 分轮独立推演                           │
│  ┌─ 推演方案A ──────────────────────────────────────────────────┐   │
│  │  Step 1: 起点动作 → ⚠️缺口检测 → 查全局补全箱 → 补全 → 继续    │   │
│  │  Step 2: 关键路径 → ⚠️缺口检测 → 查全局补全箱 → 补全 → 继续    │   │
│  │  Step 3: 终点+匹配度 → 计算 V_final = V×0.6 + M×0.4           │   │
│  └────────────────────────────────────────────────────────────────┘   │
│  (每个方案独立推演，共享全局补全箱)                                    │
└──────────────────────────┬───────────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────────────┐
│              第3阶段: 汇总比较（按 V_final 排序）                     │
│  汇总表: 原始V / 匹配度M / V_final / 方差 / 匹配分析                 │
│  UCB = V_final + 探索项 → 排序选最优                                 │
│  V_final领先不多但匹配度显著高的 → 输出建议提示                       │
└──────────────────────────┬───────────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────────────┐
│                        第4阶段: 执行选中方案                          │
│  只执行第1名的方案                                                   │
│  执行中遇到意外 → 记录TD误差 → 可选回到第2阶段重新推演               │
└──────────────────────────┬───────────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────────────┐
│              第5阶段: 事后学习(TD更新)                                │
│  收集反馈 → TD误差 → 更新知识图谱 → 持久化                           │
│  更新约束清单到知识图谱（下次类似任务可直接引用）                      │
└──────────────────────────────────────────────────────────────────────┘
```

## ⚡ 自动启动检测（关键）

本 Skill 设置为 `alwaysApply: true`，意味着每次对话都会加载。
但加载 ≠ 启动决策流程。**是否启动决策流程，由以下自动检测逻辑决定**。

### 检测时机

```
检测发生在两个时刻:

时刻1: 每次收到用户消息后
  → 判断: 用户的问题是否涉及"需要决策"?
  → 如果是 → 启动完整决策流程
  → 如果不是 → 保持静默，不干扰正常对话

时刻2: 每次工具调用返回结果后
  → 判断: 刚刚执行的结果是否表明"当前方案不行"?
  → 如果是 → 启动"再推演"模式
  → 如果不是 → 继续当前路径
```

### 时刻1的检测规则

```
收到用户消息后，执行快速检测。

核心原则: 不依赖用户的措辞（"选哪个"、"不确定"），
          而是分析任务本身的性质。

检测流程:

第一步: 任务类型识别
  用户的问题/任务属于以下哪种类型:
  
  Type A: 执行类 — "帮我做XX"、"实现XX"、"修改XX"
    → 这是一个需要执行的任务，但可能涉及多个方案
    → 需要进一步检测是否触发引擎
  
  Type B: 决策类 — "选A还是B"、"哪个方案更好"
    → 明确需要决策，直接触发引擎 ✓
  
  Type C: 信息类 — "XX是什么"、"怎么理解XX"
    → 不需要执行，不需要决策，不触发 ✗
  
  Type D: 排查类 — "为什么XX出错了"、"XX不正常"
    → 可能需要多路径排查，触发简化推演 ✓

第二步（仅Type A需要）: 执行类任务的决策需求检测
  问自己: "这个任务，有没有多种合理的实现方式?"
  
  判断依据:
  □ 任务可以用不同的技术方案实现? (比如认证可以用JWT/OAuth/Session)
  □ 任务可以在不同的位置实现? (比如校验可以在前端/后端/网关)
  □ 任务需要选择不同的工具? (比如用A库还是B库)
  □ 任务的实现方式会显著影响后续维护?
  
  判定:
  如果有 ≥2 个"是" → 这是一个有隐藏决策点的任务 → 触发引擎 ✓
  如果全部"否" → 这是一个机械执行任务 → 不触发 ✗

示例:
  用户说: "帮我实现用户认证"
    检测:
    - Type A: 执行类任务
    - 多种实现方式? → JWT/OAuth2/Session/SSO → YES ✓
    - 不同位置? → 中间件/拦截器/网关 → YES ✓
    - 不同的库? → gin-jwt/casbin/自实现 → YES ✓
    → 触发决策引擎 ✓

  用户说: "把这个按钮颜色改成蓝色"
    检测:
    - Type A: 执行类任务
    - 多种实现方式? → 改CSS/改JS/改主题 → NO（都是改颜色值）
    - 不同位置? → 只有一个按钮 → NO
    → 不触发 ✗
  
  用户说: "服务器返回500错误，帮我看看"
    检测:
    - Type D: 排查类任务
    → 触发简化推演 ✓（多路径排查）

第三步: 确认启动
  → 输出启动信号
  → 进入第0步: 需求约束收集
```

### 如何理解"检测发生在时刻1"

```
不要误解: "每次收到消息都要检测一遍，会不会很慢？"

不会。检测的逻辑是:
  1. 在你的正常推理过程中，额外做一个快速判断
  2. 这个判断不需要调用任何工具，不需要查资料
  3. 只是问自己一个问题: "这任务有多个可行的方案吗？"
  4. 有 → 启动引擎；没有 → 跳过

类比人脑:
  别人让你做一件事，你下意识就会想:
  "这个嘛...有好几种做法" → 然后开始想哪种最好
  "这个很简单，直接做就行" → 直接动手
  这个Skill要模拟的就是这个"下意识判断"。
```

### 时刻2的检测规则

```
工具调用返回后，检测执行结果:

检测项A: 执行是否遇到意外?
  编译失败? → 触发"再推演" ✓
  测试失败? → 触发"再推演" ✓
  结果与预期不符? → 触发"再推演" ✓
  一切正常? → 不触发 ✗

检测项B: 执行结果是否改变了之前的判断?
  发现了之前不知道的约束? → 触发 ✓
  某个方案被证明不可行? → 触发 ✓
  新信息让其他方案变得更优? → 触发 ✓
  没有新信息? → 不触发 ✗

触发"再推演"时:
  → 记录本次执行的TD误差
  → 回到第2阶段，对剩余方案重新推演
  → 如果当前方案仍最优 → 继续执行
  → 如果有更优方案 → 切换
```

### 启动信号

```
当自动检测决定启动决策流程时，输出以下启动信号:

"═══════════════════════════════════════
 ⚡ [MCTS-TD] 检测到决策需求，启动决策引擎
 触发原因: [检测项A/B/C的具体描述]
 启动模式: [完整推演/简化推演/再推演]
═══════════════════════════════════════"

这个信号让用户知道:
  - 引擎被触发了
  - 为什么被触发
  - 当前处于哪个阶段
```

### 不干扰原则

```
当自动检测决定"不触发"时:
  → 不输出任何与引擎相关的内容
  → 完全保持静默，像不存在一样
  → 不影响正常的对话和工作流程
  
目的: 这个 Skill 是"备用决策系统"，只在需要时启动，
      不需要时完全不干扰用户。
```

## 触发规则

> 本 Skill 模拟人脑的通用决策机制，**不限于任何特定行业或领域**。
> 触发条件基于**决策本身的特征**，而不是基于"你在做什么工作"。

### 触发特征

```
当以下条件中任意一条成立时，触发本决策引擎：

条件A: 决策不确定性高
  你对"什么是最优方案"没有把握
  多个方案看起来各有利弊
  简单判断: "我不确定哪个方案最好"

条件B: 决策后果重要
  选错了会有显著成本（时间/金钱/质量）
  一旦执行很难回退
  简单判断: "选错了会很麻烦"

条件C: 决策信息不完整
  你缺少做出判断所需的某些信息
  需要先了解一些东西才能做决定
  简单判断: "我还需要了解更多才能决定"

条件D: 方案差异化明显
  不同方案的优缺点各不相同
  没有哪个方案在所有维度上都占优
  简单判断: "方案各有千秋"
```

### 决策特征矩阵

每个任务到达时，按以下矩阵判断是否需要触发本引擎：

```
                ┌─────────────┬─────────────┬─────────────┐
                │ 后果轻微     │ 后果中等     │ 后果严重     │
├─────────────┼─────────────┼─────────────┼─────────────┤
│ 确定性强     │ 直接执行     │ 简化推演     │ 完整推演     │
│              │ (不触发)     │ (2步推演)    │ (3步推演)    │
├─────────────┼─────────────┼─────────────┼─────────────┤
│ 不确定性中   │ 直接执行     │ 简化推演     │ 完整推演     │
│              │ (不触发)     │ (2步推演)    │ (3步推演)    │
├─────────────┼─────────────┼─────────────┼─────────────┤
│ 不确定性高   │ 简化推演     │ 完整推演     │ 完整推演     │
│              │ (2步推演)    │ (3步推演)    │ (3步+多轮)   │
└─────────────┴─────────────┴─────────────┴─────────────┘

说明:
  "确定性强" = 你很清楚要做什么，方案明确
  "不确定性高" = 你不太确定，需要探索
  "后果轻微" = 选错了没什么大影响
  "后果严重" = 选错了代价很大
```

### 推演深度定义

| 模式 | 触发条件 | 执行内容 |
|------|---------|---------|
| **不触发** | 确定性强 + 后果轻微 | 直接执行，不走决策流程 |
| **简化推演** | 后果中等或确定性中等 | 约束收集 → 方案枚举 → 2步推演 → 汇总 → 执行 |
| **完整推演** | 后果严重或不确定性高 | 完整6步: 约束→领域认知→方案生成→3步推演→汇总→执行 |
| **深度推演** | 后果严重 + 不确定性高 | 完整推演 + 多轮MCTS模拟 + 多视角交叉验证 |

### 适用领域示例（非穷举）

```
由于本引擎是通用决策框架，适用于任何需要"先想清楚再做"的场景：

▸ 技术领域:
  方案选择: 技术选型、架构设计、工具选择
  问题排查: 故障定位、性能优化、安全审计
  规划类: 重构计划、迁移方案、测试策略

▸ 非技术领域（思路相通，概念可映射）:
  规划类: 旅行路线规划、学习路径设计、投资方案比较
  选择类: 产品选型、供应商评估、方案竞标
  排查类: 流程优化、根因分析、风险评估

无论什么领域，核心决策逻辑是一样的：
  收集约束 → 理解领域 → 生成方案 → 逐一推演 → 比较选优 → 执行 → 学习
```

---

### 模式一：完整模式（默认）

```
枚举方案(限5个) → 逐轮推演(每方案1轮) → 汇总比较(CLT-UCB) → 执行 → TD更新
```

### 模式二：快速推演（方案数 > 5）

```
直觉粗筛 → 保留top-3 → 逐轮推演 → 汇总比较 → 执行
```

### 模式三：再推演（执行途中遇到意外）

```
执行受阻 → 记录TD误差 → 回到第2阶段重新推演剩余方案 → 切换执行
```
