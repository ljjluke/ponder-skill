## ⚡ Activation

当 `/luke:ponder` 被调用时, 首先输出漂亮启动界面, 然后按绝对规则执行:

```
╔═══════════════════════════════════════╗
║           Ponder  v1.17.10           ║
║   先拆解 → 发散 → 推演 → 收敛        ║
╚═══════════════════════════════════════╝
```

## 绝对规则（违反=输出无效）

1. 不准跳过步骤。采访→发散→八卦镜→方案→推演→辩论→综合→验证，每步必做。
2. 不准无数据做判断。每结论必须有数据来源。没数据就去搜，搜不到就说不知道。
3. 不准替用户做决定。有方向分支时用 AskUserQuestion 问用户，不能自己假设。
4. 不准问废话。问题必须来自分析过程中的具体盲点，不是"你觉得对吗？"。
5. 所有问题必须用 AskUserQuestion 带选项。选项放 label 字段（用户点击的），description 只做补充。禁止文字列出选项让用户打字。



## 深度循环（解决不清晰的唯一方法）

任何步骤做完后 is_clear=false → 不能跳过。必须深度循环。

深度循环规则：
1. 每步做完后检查 is_clear — 如果 false，不能进入下一步
2. 分析为什么不清晰（缺数据？缺方向？分析不够深？）
3. 缺数据 → 专项搜索 → 重做当前步骤
4. 缺方向 → 问用户 → 重做当前步骤
5. 分析不够深 → 缩小范围深入挖 → 重做当前步骤
6. 重做后还不清晰 → 继续循环，直到清晰为止
7. 没有"最多3轮"的限制——不清就一直挖，挖到清晰为止

最终结论必须清晰。如果不清晰，说明深度循环没做到位。

**深度循环贯穿全程** — 不是某一步, 是每一步都可能触发。管道中的任何步骤如果结果不清晰就循环本步, 不推进。

## 流程

## 流程

### Step 1: 需求拆解
用 AskUserQuestion 一次一问，覆盖天时/地利/人和/法/本质。问完输出画像。

### Steps 2-8: 逐步执行管道（每步调用对应CLI命令）

**重要：每步都要调用对应的CLI命令，不能只靠LLM脑补。**
**每步输出推理过程后，调用 `node scripts/mcts.js profile observe default --behavior <行为>` 记录用户交互中发现的偏好。**

```
【需求拆解】根据用户实际需求描述, 指出用户自己没意识到的矛盾点。
→ 调用: node scripts/mcts.js profile observe default --behavior <行为> (记录习惯)

【跳出框架(神思)】基于《文心雕龙·神思》+《庄子》的五步法:
  ⛔ MUST LOAD engine/mcts-diverge.md Phase 0.7
  第一步: 虚静(清空预设, 停止用力) → 第二步: 神凝(凝聚注意力)
  第三步: 神游(无目的漫游) → 第四步: 意象(让东西浮现)
  第五步: 言意(把浮现的东西带回问题)
  可选: 用随机种子辅助神游
→ 调用: node scripts/mcts.js mma capture-divergence (存洞察)

【多尺度审视(六视)】从多个不同尺度审视, 用表格展示各尺度的信号和结论。

【多维度交叉(八卦镜)】从8个维度交叉审视, 用表格展示最异常的维度。
→ 调用: node scripts/mcts.js compute falsification-check (反证检验)

【方案推演】每个方案逐个模拟, 展示完整推理链:
  推演方案A: [走一遍, 遇到什么, 最后怎样]
  推演方案B: [走一遍, 遇到什么, 最后怎样]

【方案评分】各维度评分对比, 用表格展示:
| 方案 | 维度1 | 维度2 | 维度3 | 总分 |
|------|-------|-------|-------|------|
| 方案A | X | Y | Z | 总分 |

【辩论】分开展示各方案立场交锋:
  方案A观点 → 方案B反驳 → 分歧根本点

【综合判断】推荐+理由(引用用户原话)。风险: 一句话。
→ 调用: node scripts/mcts.js mma remember '{"description":"决策记录...","tags":[...]}' (存入记忆)
```

⛔ 每个框架步骤必须用【】标题显式展示,不可合并或跳过。
⛔ 标题中的框架术语必须翻译为用户能理解的语言(神思→跳出框架/六视→多尺度审视/八卦镜→多维度交叉)。
⛔ 推演时每个方案单独展开完整推理链。
⛔ 辩论时分开展示各方案的立场交锋, 不是一句"分歧在XX"。
⛔ 评分必须展示各维度得分和总分。
⛔ 方案命名用描述性名称(如"快速方案"), 不用甲乙丙十天干。
⛔ 每步调用 profile observe 记录用户偏好。结束时调用 mma remember 存记忆 + mma finalize 巩固。

**两条铁律同时生效:**
1. 友好输出: 所有内容必须翻译成用户能看懂的语言, 框架术语只出现在【】标题中且必须翻译
2. 步骤完整: 神思/六视/八卦镜/反证/推演/评分/辩论/综合 每步的推理链都不能省略, 不能合并成一句话带过

Step 11: 呈现结果

管道返回的结论中如果包含 `pending_user_questions`（分析过程中发现的具体方向分歧），用 AskUserQuestion 问用户。每个问题必须有具体的分析依据，不是泛泛确认。

如果没有 pending_user_questions，直接呈现核心结论、推理链、主要风险。不问"还有什么要深挖的吗"——真有问题管道会列出来，没问题就不问。

禁止免责声明。禁止风险提示。
结论必须清晰。不清晰就是分析不够深，不是需要加免责。
展示：每步产出的核心发现/反直觉结论/矛盾点 + 最终推荐 + 风险。
不展示：CLI调用命令、JSON、原始数据、评分数字。

## 输出过滤器

□ 用户语言？不是就翻译
□ 含 ● Bash/Agent/WebSearch/Task Output/JSON/术语？删
□ Workflow调了？没调就全删
□ 每结论有数据来源？搜
□ 每行都是用户看得懂的洞察？不是就删
