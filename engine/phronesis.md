## 行动翻译 — 把结论翻译成可执行的动作 (Action Translation)

**学术来源**: 亚里士多德「实践智慧」(phronesis) + 杜威「手段-目的连续体」(means-ends continuum)。

### 为什么要这一步

综合阶段产出的结论是 episteme 层——它说"这是最优方案"、"风险是X"、"可谬点是Y"，但没说"现在该做哪个具体动作"。从"知道该选什么方案"到"知道该做什么动作"，中间差了一步翻译。

行动翻译层的职责是：**把综合阶段的结论翻译成具体、可执行、带判据和回退条件的行动步骤。**

这个层解决一个结构性缺陷：框架输出"建议做X"但没有说怎么做——用户拿着结论不知道怎么落地。没有翻译层，结论是纸面上的。

### 哲学底座

- **亚里士多德「实践智慧」**: 不是理论知识(episteme，"这是什么情况")，不是技艺(techne，"怎么做")，而是在具体情境中判断"该做什么"的能力。phronesis 补的就是 episteme 到行动之间的翻译。
- **杜威「手段-目的连续体」**: 手段和目的不是分离的。目的不是固定终点，是在行动中被不断重定义的。行动步骤不是从结论机械推导的，是在结论的约束下、根据当前情境动态生成的。

### 门控

```
① 高赌注问题（不可逆/高成本/涉及他人/长期影响）→ 必做行动翻译
② 可逆小事 → 跳过（输出"结论已可直接执行"即可，无需翻译）
```

### 操作：行动翻译三步

```
① 行动生成(action_plan)：把"推荐方案X"翻译为具体可执行步骤序列——
   - 第一步是什么？（明天谁做什么、用什么工具、花多少资源）
   - 第二步依赖第一步的什么结果？（不是"然后做Y"而是"如果第一步得到A则做Y，得到B则做Z"）
   - 每步的完成判据是什么？（做完的标志不是"做完了"，是"X指标从A变成B"）
   ⛔ 不是方案阶段终态画像的复读——终态画像是方向性的"成了是什么样"，
   行动翻译是操作性的"今天下午做这件事"。

② 回退条件(kill_conditions)：什么信号/结果出现时必须中止行动——
   - 基于推演的翻车点：推演说X在第三个月翻车 → 如果第一步做完X的前置条件就不成立了 → 中止
   - 基于可谬标注：可谬说"若X不成立结论就错" → 如果执行中发现X不成立 → 中止
   ⛔ 必须给出至少一条可观测的中止条件，不接受"视情况而定"。

③ 结果预期(expected_results)：执行完成后什么算"做对了"——
   - 不是"方案目标达成"（太远），是"第一步/第二步做完后，能观察到什么具体变化"
   - 预期结果必须可被传感器验证——传感器能检测到才算有效预期
   - 预期结果直接喂给结果学习层做 outcome_gap 对比
```

### 产出格式

行动翻译的产出是综合结论的 action_translation 字段：

```json
{
  "action_translation": {
    "urgency": "立即/本周/观察",
    "action_plan": [
      {"step": 1, "action": "具体动作", "depends_on": "依赖的上一结果", "criterion": "完成判据"}
    ],
    "kill_conditions": ["中止条件1", "中止条件2"],
    "expected_results": [
      {"metric": "可观测指标", "expected": "预期值", "sensor": "哪个传感器能验证"}
    ]
  }
}
```

### 与已有模块的边界

- **行动翻译 vs 方案阶段终态画像**: 终态画像是"成了是什么样"（方向性），行动翻译是"今天做什么"（操作性）。终态画像在方案阶段产，行动翻译在综合阶段后产——两者是同一个目的的不同分辨率。
- **行动翻译 vs 综合结论**: 综合结论是"推荐方案X，风险Y"，行动翻译是"推荐方案X的第一步是A，如果第一步结果偏离预期B则中止"。行动翻译不重写结论，只给结论加上操作层。
- **行动翻译 vs 结果学习**: 行动翻译的 expected_results 是结果学习的输入——执行结果回来后，对比 expected_results 和实际结果，判断判断被验证还是证伪。
- **行动翻译 vs 传感器**: 行动翻译不实现传感器，只定义"什么结果可由哪个传感器验证"——传感器是外部能力，框架只对接格式。
