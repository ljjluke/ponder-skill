## 结果学习 — 行动结果回流后的判断验证与修正 (Outcome Learning)

**学术来源**: 皮尔士「探究的连续性」(inquiry as continuous) + 贝特森(Bateson)「学习层级」(levels of learning)。

### 为什么要这一步

框架做完判断、翻译成行动、行动被执行——但这之后呢？执行结果回来了，框架需要知道"我判断对了吗"、"如果错了，错在哪"、"为什么这类问题上我反复错"。

当前 evolve.js 只看"管道是否清晰"不看"判断是否正确"——它关心的是推理过程的流畅度，不是判断结论的准确度。结果学习层补的就是这个"判断对不对"的回流。

结果学习层的职责是：**执行结果回流后，对比预期和实际、修正立场、修正推理模式。**

没有学习层，做对了做错了都白做——框架每次推理都是独立的新生命，从上次结果里学不到任何东西。

### 哲学底座

- **皮尔士「探究的连续性」**: 探究不是一锤子买卖。"一个结论只是此刻最可信的假设"——上次结论被证伪了不是失败，是探究的下一轮开始。
- **贝特森「学习层级」**: 零阶学习（对刺激做固定反应）、一阶学习（改反应策略）、二阶学习（改学习策略本身）。当前框架的一阶学习（working_stance 因证据改变）已有一阶机制，二阶学习（"我总在某类判断上翻车 → 改推理方式"）缺。

### 门控

```
① 有执行结果回流（传感器检测到行动翻译的 expected_results 有了实际值）→ 启用结果学习
② 无可回流传执行结果 → 跳过（基于推理结果本身做立场修正的方式已有 evolve/finalize）
```

### 操作：结果学习三步

```
① 结果对比(outcome_gap)：执行实际结果 vs 行动翻译的 expected_results——
   - 实际符合预期 → 记录"判断验证通过"（强化当前立场和推理模式）
   - 实际偏离预期 → 记录"判断被证伪"（触发修正）
   - 实际无法判定（信号不够/时间不够）→ 记录"待继续观察"
   ⛔ 不是杜威实验循环的复读——dewey_loop 是行动前的模拟验证（在推演阶段），
   结果学习是行动后的真实验证。模拟验证和真实验证是两回事。

② 立场修正(stance_update)：如果执行结果证伪了综合立场——
   - 更新 working_stance（立场被真实行动后果修正 → 持久化到 framework_self）
   - 更新用户画像的约束/假设（执行暴露了画像里没有的真实约束）
   - 记录"上次在这个问题上的立场被证伪了，原因是X"（喂给信号过滤层）

③ 推理模式修正(pattern_learning)：如果同类型问题反复翻车在同一类判断上——
   - 二阶观察的 pattern_blindspot 从"单次推理的觉察"升级为"持续约束"
   - 写入 evolve-rules.json：下次同类问题在对应步骤加检查点
   ⛔ 这是贝特森二阶学习的落地——不是改反应策略，是改学习策略本身
```

### 数据流

```
行动执行 → 传感器检测结果 → 结果学习层
  ↓
outcome_gap 对比 expected_results vs 实际
  ↓
validated → 强化立场 → framework_self.stance_memory 标记 validated
falsified  → 修正立场 → framework_self.stance_memory 标记 falsified + 原因
           → 更新画像 → user_profile 更新暴露的真实约束
           → 模式学习 → framework_self.pattern_memory 积累
pending    → 标记待观察 → framework_self.stance_memory 标记 pending
```

### 与已有模块的边界

- **结果学习 vs evolve.js**: evolve.js 看管道流畅度（步骤清晰率、问题数），结果学习看判断准确度（预期 vs 实际）。两者互补：evolve.js 优化"推理怎么跑"，结果学习优化"判断对不对"。
- **结果学习 vs working_stance**: 立场在单次推理内的演化（收敛→辩论→综合）是"推理中的改主意"；结果学习是"执行后的事实打脸"——两者都是立场修正，但依据不同（推理内 vs 执行结果）。
- **结果学习 vs 结论自反**: 结论自反是行动前质疑共享前提（"我们共享的前提为什么不成立"），结果学习是行动后对比（"前提真的成立了吗"）。
