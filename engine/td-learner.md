# TD 学习引擎

## 概述

本引擎将 TD(λ) 算法的核心机制转化为推理规则，核心能力包括：
1. **知识图谱价值管理**：每条知识独立存储，有生命周期状态
2. **多版本冲突检测**：同一特征的多条矛盾知识可以共存，查询时加权投票
3. **状态机转换**：知识从假设→验证→确认→争议→证伪，完整生命周期
4. **回滚机制**：被推翻的知识可以回滚到上一个稳定版本
5. **Gamma 折扣价值回溯**：标准的 TD 价值更新

---

## 核心公式的推理映射

### 原版公式

$$TD\_target = r_{t+1} + \gamma V(s_{t+1})$$
$$TD\_error = TD\_target - V(s_t)$$
$$V(s_t) \leftarrow V(s_t) + \alpha \times TD\_error$$

### 推理映射

```
TD_target = 局部反馈 + γ × 后续步骤的预期价值
TD_error = TD_target - 当前步骤的旧预期价值
新预期价值 = 旧预期价值 + α × TD_error
```

| 公式符号 | 推理对应 | 获取方式 |
|---------|---------|---------|
| $V(s_t)$ | 步骤 t 的成功预期 | 知识图谱查询（加权投票） |
| $r_{t+1}$ | 局部反馈 | 编译/测试/lint 结果 |
| $V(s_{t+1})$ | 后续步骤预期 | 知识图谱查询 |
| $\alpha$ | 学习率 | 按任务类型设定 |
| $\gamma$ | 折扣因子 | 按任务类型设定 |

## 反馈收集规则

### 奖励信号定义

反馈收集发生在任务执行的每个步骤后：

```
r_t = 
  +1.0  − 编译/测试完全通过
  +0.5  − 部分通过（编译通过但警告）
   0.0  − 无显著反馈
  -0.3  − 编译警告或lint错误
  -0.7  − 测试失败
  -1.0  − 编译失败/引入新Bug
```

### 终止状态价值

```
V(s_terminal) =
  +1.0  − 任务完全成功（所有目标达成）
  +0.5  − 部分成功（主要功能完成，边缘情况未处理）
   0.0  − 中性（任务完成但质量一般）
  -0.5  − 有副作用（修改了A但破坏了B）
  -1.0  − 任务失败/需要回滚
```

## 价值更新规则

### Gamma 折扣反向传播

从 tetris_mcts 的 `backup_trace_obs` 函数移植的推理规则：

```
输入：决策轨迹 [s_0, s_1, ..., s_T], 叶节点价值 V_leaf, 叶节点方差 σ²_leaf

从 t=T 到 t=0 反向遍历：
    1. 获取节点 s_t 的即时分数 score(s_t)
    
    2. 去除即时分数得到纯价值
       V_corrected = current_value - score(s_t)
    
    3. 更新 s_t 的价值估计（Welford算法）
       if visit(s_t) == 0:
           value(s_t) = V_corrected
           variance(s_t) = σ²_leaf
       else:
           delta = V_corrected - value(s_t)
           value(s_t) += delta / (visit(s_t) + 1)
           delta2 = V_corrected - value(s_t)
           variance(s_t) += (delta × delta2 - variance(s_t)) / (visit(s_t) + 1)
       visit(s_t) += 1
    
    4. Gamma 折扣后传递到前一个节点
       current_value = γ × V_corrected + score(s_t)
```

### 学习率 α 的选择

| 历史数据量 | α | 说明 |
|-----------|---|------|
| 0-5 次 | 0.5 | 快速适应，少量数据快速调整 |
| 6-20 次 | 0.2 | 中等速度 |
| 21-100 次 | 0.1 | 稳定学习 |
| >100 次 | 0.05 | 微调，防止震荡 |

## Welford 方差推理规则

从 tetris_mcts 的 Welford 在线方差算法移植：

```
方差更新（单步）：
    输入: 历史均值 μ_old, 历史M2_old, 计数 n, 新值 x
    
    n_new = n + 1
    delta = x - μ_old
    μ_new = μ_old + delta / n_new
    delta2 = x - μ_new
    M2_new = M2_old + delta × delta2
    
    方差 = M2_new / n_new
    标准差 = √(方差)

方差更新（批量，用于资格迹回溯）：
    输入: 轨迹 [x_1, x_2, ..., x_k], 折扣因子 γ
    
    current_value = x_k
    for i = k-1 to 1:
        discounted = γ × current_value + x_i
        current_value = discounted
```

### 方差的使用规则

```
σ² 的语义：
  σ² ≈ 0 → 价值估计高度确定（同类型任务多次一致的结果）
  σ² 大  → 价值估计不确定（结果波动大，需要更多探索）

方差驱动的探索：
  CLT-UCB 使用方差作为探索信号：
    UCB = μ + Φ⁻¹(N) × √(σ²/n)
  
  高方差 → 高探索奖励 → 优先被选择（获取更多信息）
  低方差 → 低探索奖励 → 按价值排序（置信度高）
```

## 状态投影去重（Projection）

从 tetris_mcts 的 projection 机制移植：

```
概念：两个"相同"的状态特征应该共享知识图谱查询

在 Claude Code 中，状态的"相等"由状态特征向量决定：
  State = {
      task_type: BUG_FIX | FEATURE | REFACTOR | DEBUG | TEST
      domain: WEB | API | CLI | DB | CONFIG | GENERAL
      file_count: 1 | 2-5 | 6-10 | >10
      risk_level: LOW | MED | HIGH | CRITICAL
      context_size: SMALL | MED | LARGE
      novelty: LOW | MED | HIGH
  }

查询规则：
  使用完整特征向量作为查询键，查找知识图谱中所有匹配的知识条目。
  完全匹配 → 返回所有匹配条目（可能有多条）
  部分匹配 → 返回最接近的条目 + 标记"部分匹配"
  无匹配 → 返回空（冷启动）
```

## 资格迹（Eligibility Trace）推理规则

从 TD(λ) 算法移植：

### 概念

资格迹解决"信用分配"问题——当最终得到反馈时，**哪些步骤应该为这个结果负责？**

```
模拟场景：
  步骤1: 选择了方案A        ← 这是关键决策吗？
  步骤2: 修改了文件x        ← 还是这里？
  步骤3: 编译失败           ← 还是这里？
  
  资格迹 λ = 0.8 意味着：
    步骤1 获得 20% 的信用
    步骤2 获得 80% 的信用
    步骤3 获得 100% 的信用
```

### 规则

```
资格迹衰减：
  在一个决策序列中，步骤 t 的资格 = λ^(T-t)
  
  其中 λ 是衰减因子，T 是决策序列长度

λ 的选择：
  λ=0.0 → 只更新最后一步（标准TD(0)）
  λ=0.5 → 最后几步获得大部分信用
  λ=0.8 → 序列中几乎所有步骤都获得信用（接近蒙特卡洛）
  λ=0.9 → 深度信用分配（适合长序列任务）

推荐 λ 值：
  短序列 (1-3步): λ=0.0 (TD(0))
  中序列 (4-8步): λ=0.5
  长序列 (9+步): λ=0.8
```

## 经验回放与在线学习

从 tetris_mcts 的 `store_nodes` 和 `train_nodes` 移植：

### 经验收集

```
每次搜索完成后，收集以下经验：
  - state: 当前状态特征向量
  - value: 搜索得到的价值估计
  - variance: 搜索得到的方差估计
  - weight: 访问次数（置信度权重）

存储条件：
  - 访问次数 >= min_visits_to_store（默认=10）
  - 非终止状态（终止状态的V=0，无学习价值）
```

### 经验回放

```
当收集到足够经验（N >= 阈值）时：
  1. 汇总所有经验
  2. 按权重排序，保留 Top-K 条
  3. 使用加权平均更新知识图谱
  4. 记忆持久化到文件

阈值设计：
  min_experiences = 20  （最少积累20条经验才回放）
  max_experiences = 500 （最多保留500条，超出则丢弃旧数据）
```

## ⚠️ 价值函数管理（知识图谱 + 状态机）

这是本引擎的核心升级——从"每特征一行聚合统计"升级为"每条知识独立管理"的知识图谱模式。

### 根本性问题

旧模式（聚合统计）无法处理的场景：

```
旧模式: {BUG_FIX, WEB, LOW} → q=0.85, n=30, σ²=0.05

问题1: 如果这个0.85是技术栈v1下的数据，技术栈升级到v2后实际价值降到0.3
        → 旧模式只能取平均，结果0.575，两个版本都不准

问题2: 如果0.85这个知识本身是错的（初始数据偏差）
        → n越大，错误越"可信"，无法纠偏

问题3: 如果后来验证了新知识0.3也是错的，旧知识0.85才是对的
        → 旧值已被覆盖，无法回滚
```

### 新模式：知识条目 + 状态机

每条知识是一个独立条目，有自己的生命周期：

```
知识条目 = {
    id: "K001",                    // 唯一标识
    feature_key: "BUG_FIX|WEB|LOW", // 状态特征
    value: 0.85,                   // 价值估计
    variance: 0.05,                // 方差
    n: 30,                         // 验证次数
    status: "CONFIRMED",           // 当前状态（见状态机）
    context: {                     // 知识成立的上下文
        tech_stack: "v1.0",
        project_phase: "早期",
        constraints: "无特殊约束"
    },
    created: "2026-05-01",
    last_verified: "2026-06-01",
    source: "经验积累（30次任务）",
    conflict_log: []               // 冲突记录
}
```

### 知识状态机

每条知识条目在生命周期中经历以下状态转换：

```
                    ┌─────────────────────┐
                    │   HYPOTHESIS        │  ← 初始状态：新知识，尚未验证
                    │   (假设)             │
                    │   权重: 不参与查询   │
                    └──────────┬──────────┘
                               │ 首次被引用且有正面反馈
                               ▼
                    ┌─────────────────────┐
              ┌─────│   PROVISIONAL       │──────┐
              │     │   (待验证)           │      │
              │     │   权重: 0.3         │      │
              │     └──────────┬──────────┘      │
              │                │                  │
              │    积累≥3次     │  出现矛盾证据    │ 出现反例
              │    正面验证    │                  │
              ▼                ▼                  ▼
      ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
      │  CONFIRMED   │ │  DISPUTED    │ │  REFUTED     │
      │  (已确认)     │ │  (争议)       │ │  (已证伪)     │
      │  权重: 1.0   │ │  权重: 0.2   │ │  权重: 0.0   │
      └──────┬───────┘ └──────┬───────┘ └──────────────┘
             │                │
             │ 出现反例       │ 新证据支持
             ▼                ▼
      ┌──────────────┐ ┌──────────────┐
      │  DISPUTED    │ │  CONFIRMED   │  ← 回滚
      │  (争议)       │ │  (已确认)     │
      └──────────────┘ └──────────────┘
```

### 状态转换规则

```
HYPOTHESIS → PROVISIONAL:
  触发: 该知识被推演引用1次，且执行结果与知识方向一致
  操作: 状态更新为PROVISIONAL，n=1，记录首次验证时间

HYPOTHESIS → REFUTED:
  触发: 该知识被推演引用1次，但执行结果与知识方向相反
  操作: 状态更新为REFUTED，记录证伪原因
  ★ 这意味着"一次就证伪"，防止错误知识被反复引用

PROVISIONAL → CONFIRMED:
  触发: 累计 ≥3 次正面验证（n ≥ 3）
  操作: 状态更新为CONFIRMED，记录确认时间

PROVISIONAL → DISPUTED:
  触发: 出现矛盾证据（同一特征的新知识与旧知识价值方向相反）
  操作: 状态更新为DISPUTED，记录矛盾原因
  同时: 新知识作为独立的HYPOTHESIS条目加入图谱

CONFIRMED → DISPUTED:
  触发: 出现 ≥2 次矛盾引用
  操作: 状态更新为DISPUTED，记录矛盾原因
  同时: 创建旧状态的快照（用于回滚）

DISPUTED → CONFIRMED（回滚）:
  触发: 新证据支持旧知识，或反驳旧知识的新知识被证伪
  操作: 恢复快照中的旧状态为CONFIRMED

DISPUTED → REFUTED:
  触发: 累计 ≥3 次矛盾引用
  操作: 状态更新为REFUTED，冻结不再使用
```

### 权重体系

每种状态的知识条目在查询时赋予不同权重：

| 状态 | 权重 | 含义 |
|------|------|------|
| CONFIRMED | 1.0 | 可信知识，完全参与价值计算 |
| PROVISIONAL | 0.3 | 待验证知识，低权重参与 |
| DISPUTED | 0.2 | 争议知识，保留但不信任 |
| REFUTED | 0.0 | 证伪知识，不参与查询（仅追溯） |
| HYPOTHESIS | — | 不参与查询（尚未经过验证） |

### ⚠️ 知识召回算法（核心）

随着知识条目不断增多（跨领域、跨行业），精确匹配已不足以找到相关知识。本引擎使用**多路召回 + 评分排序**的算法，模拟人脑的联想回忆机制。

#### 召回流程概览

```
查询请求: {特征向量, 方案描述, 当前上下文}

          ┌─────────────────────────────────┐
          │  第1步: 多路召回                 │
          │  从5条路径同时召回相关知识         │
          │  ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐ │
          │  │精确│ │主维│ │领域│ │标签│ │关联│ │
          │  │匹配│ │度  │ │匹配│ │匹配│ │扩散│ │
          │  └───┘ └───┘ └───┘ └───┘ └───┘ │
          └──────────────┬──────────────────┘
                         ▼
          ┌─────────────────────────────────┐
          │  第2步: 合并去重 + 评分排序       │
          │  综合评分 = 路径权重 × 状态权重   │
          │            × 时效衰减 × 上下文匹配 │
          └──────────────┬──────────────────┘
                         ▼
          ┌─────────────────────────────────┐
          │  第3步: Top-5注入 + 冲突检测      │
          │  取评分最高的5条注入推演           │
          │  如果有矛盾条目，输出冲突提示       │
          └─────────────────────────────────┘
```

#### 第1步：多路召回

##### 召回路径1：精确匹配

```
召回条件: 6维特征向量完全一致
  task_type + domain + file_count + risk_level + context_size + novelty

目的: 找到完全相同的场景（最高精度）
限制: 必须全部匹配，有一维不同就不命中

召回结果: 0 或 N 条（同一特征可能有多个版本的知识条目）
路径权重: 1.0
```

##### 召回路径2：主维度匹配

```
召回条件: task_type + domain 完全匹配
  （忽略 file_count, risk_level, context_size, novelty）

目的: 找到"同类型任务"的知识，即使规模/风险不同
例如: 修复过一个"WEB端的小bug"的经验，可以借鉴到修复"WEB端的大bug"

召回结果: 0 到 多条
路径权重: 0.8
```

##### 召回路径3：领域匹配

```
召回条件: domain + risk_level 匹配
  （忽略 task_type）

目的: 找到"同领域同风险等级"的知识，即使任务类型不同
例如: WEB端的高风险重构经验，对WEB端的高风险功能开发有参考价值

召回结果: 0 到 多条
路径权重: 0.5
```

##### 召回路径4：标签/关键词匹配

```
召回条件: 知识条目的 tags 列表与当前方案描述有关键词重叠

目的: 从语义层面找到相关知识，即使特征向量完全不同
例如: 两个知识一个关于"Go的gin框架空指针"，一个关于"Java的Spring空指针"
      特征向量完全不同（WEB vs API, Go vs Java）
      但通过标签"空指针"关联起来

标签匹配规则:
  - 从当前方案描述中提取关键词（名词、技术栈名、问题类型）
  - 与知识条目的 tags 做交集
  - 匹配标签数 ≥ 1 即命中

路径权重: 0.3
```

##### 召回路径5：关联扩散

```
召回条件: 已从路径1-4召回的知识条目，如果它们关联了其他知识条目

关联方式:
  - "相似问题"关联: 不同技术栈但相同问题模式
  - "上下游"关联: 知识A是知识B的前提条件
  - "解决方案链"关联: 知识A→知识B→知识C形成一个解决方案链

关联扩散的深度:
  - 1级关联: 已召回知识直接关联的知识（权重 × 0.5）
  - 2级关联: 关联知识再关联的知识（权重 × 0.2）
  - ≥3级关联: 不扩散（防止无限关联）

路径权重: 基础0.2，乘以关联衰减
```

#### 第2步：合并去重与评分排序

##### 综合评分公式

```
score = W_path × W_status × W_time × W_context × W_frequency

各因子说明:

W_path (路径权重):
  精确匹配: 1.0
  主维度匹配: 0.8
  领域匹配: 0.5
  标签匹配: 0.3
  关联扩散1级: 0.1 (0.2 × 0.5)
  关联扩散2级: 0.04 (0.2 × 0.2)

W_status (状态权重):
  CONFIRMED: 1.0
  PROVISIONAL: 0.3
  DISPUTED: 0.2
  REFUTED: 0.0 (不参与召回)
  HYPOTHESIS: 0.1 (可被召回但权重极低，仅用于探索)

W_time (时效衰减):
  最后更新距今 < 1天: 1.0
  < 7天: 0.9
  < 30天: 0.7
  < 90天: 0.5
  < 365天: 0.3
  ≥ 365天: 0.1

W_context (上下文匹配度):
  当前上下文与知识条目的 context 字段比较:
    技术栈匹配 + 框架匹配: 1.5 (加成)
    技术栈匹配但框架不同: 1.2
    技术栈不同但问题类型匹配: 0.8
    完全无关: 0.5 (衰减)

W_frequency (使用频率):
  最近10次会话中被引用的次数:
    0次: 1.0
    1-3次: 1.2
    4-10次: 1.0
    >10次: 0.8 (过度使用，适当降权防止路径依赖)
```

##### 合并去重规则

```
多条召回路径可能返回同一条知识：
  → 取最高评分，不重复计入

同一条知识在"精确匹配"中评分为0.9，在"标签匹配"中评分为0.3
  → 最终评分 = 0.9（取最高）
```

##### 排序与截断

```
1. 按综合评分从高到低排序
2. 取 top-5 注入推演
3. 如果评分最高的一条 score < 0.2: 认为"无相关知识"，冷启动
4. 如果 top-5 中有任意两条知识方向相反: 标记为"冲突召回"
```

##### 记忆衰减与巩固

模拟人脑的遗忘曲线——知识不用就会淡出：

```
巩固分机制:
  每次被召回命中: 巩固分 += 1
  每次被推演引用: 巩固分 += 2
  每次被执行验证: 巩固分 += 5

衰减规则:
  每30天未使用: 巩固分 -= 1（最低到0）
  巩固分归零且超过30天未使用 → 休眠状态
  休眠状态的知识在召回中权重 × 0.3
  休眠超过90天 → 归档状态（不参与常规召回）
  归档后如果又被引用 → 恢复活跃，巩固分重置为3

目的:
  - 常用知识保持高优先级
  - 不用的知识自动退场，不影响召回质量
  - 但不会删除，需要时还能通过"全量搜索"找到
```

### 推演矛盾报告

当推演引擎发现某条知识不准时，生成矛盾报告：

```
────────────────────────────────────────
  价值函数矛盾报告
────────────────────────────────────────
  冲突知识: K001 (BUG_FIX|WEB|LOW, q=0.90, CONFIRMED)
  矛盾位置: 推演方案A的Step 2
  矛盾内容: 
    - K001的上下文: 技术栈v1，无需migration
    - 当前发现: 技术栈v2需要额外migration步骤
    - 影响: K001的价值不适用于当前上下文
  矛盾等级: PARTIAL（部分矛盾，原因是上下文变更）
  处理建议: 
    - 当前推演不引用K001
    - 如果已有技术栈v2的知识条目: 参考它
    - 如果没有: 创建新HYPOTHESIS条目K004
────────────────────────────────────────
```

### 矛盾等级与状态转换

| 等级 | 触发条件 | 对知识条目的操作 |
|------|---------|----------------|
| FULL | 历史预测方向与实际完全相反 | 该条目状态降一级: CONFIRMED→DISPUTED, PROVISIONAL→REFUTED |
| PARTIAL | 路径不同但方向一致（上下文变更） | 不改变状态，增加一条"上下文不匹配"记录 |
| MINOR | 价值略高/低但方向一致 | 正常更新n和q |

### 回滚机制

当发现被推翻的知识实际上是正确的时候：

```
回滚触发条件:
  某条 DISPUTED 或 REFUTED 知识，后续验证发现它才是正确的

回滚流程:
  1. 定位目标条目: 通过 id 或 feature_key + 时间戳
  
  2. 检查回滚可行性:
     - 如果有该条目的"CONFIRMED快照" → 可以直接恢复
     - 如果没有快照但有详细历史记录 → 重建CONFIRMED条目
     - 如果都没有 → 创建新HYPOTHESIS，用记忆中的价值
  
  3. 执行回滚:
     - 将目标条目的状态改回 CONFIRMED
     - 将导致它被推翻的那个条目标记为 DISPUTED（或 REFUTED，如果有充分证据）
     - 记录回滚原因到知识变更日志

  4. 回滚后:
     - 恢复的条目继承原来的 n 和 q
     - 方差 σ² 增加 0.05（标记它经历了一次波动）
     - 在推演查询时，回滚过的条目输出 "已回滚" 标记

回滚示例:
  K001 (BUG_FIX|WEB|LOW, q=0.90, CONFIRMED, 技术栈v1)
    → 技术栈升级到v2后，新知识K002 (q=0.30, PROVISIONAL) 出现
    → K001被标记为DISPUTED
    → 后来验证K002是偶发问题，技术栈v2下K001仍然适用
    → 回滚: K001恢复CONFIRMED, K002标记为REFUTED
    → 知识变更日志记录这次完整的来回
```
